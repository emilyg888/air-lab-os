air-lab-os — domain-agnostic engine
The engine never knows it's doing fraud detection. It knows: run a pattern, get a score, compare, improve.
Plug 1 — use case (patterns + detectors)
fraud detection marketing attribution credit risk + any domain
Plug 2 — dataset
bb_datasets (DuckDB) external-datasets (parquet) + any labelled data
Plug 3 — scoring policy (locked yaml)
fraud: f1 40% + expl 25% marketing: conversion % + domain-specific weights

What changes in the dataset layer redesign — now that we understand air-lab-os
Dataset as a plugin contract
air-lab-os doesn't care if data comes from DuckDB, parquet, or CSV. It calls DatasetHandle.eval_df() and gets a DataFrame back. The plugin provides the handle. The engine never knows the source.
Scoring policy is per use case
Each use case brings its own locked scoring policy YAML. Fraud uses f1 + explainability + latency. Marketing might use conversion rate + cost. The evaluator reads whichever policy the use case provides.
Pattern tiers mirror dataset tiers
patterns/scratch runs on bronze data. patterns/working competes on silver. patterns/stable is certified on gold. Both axes compound together.
